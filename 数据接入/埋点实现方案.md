## 二、Google Analytics 埋点体系

### 2.1 初始化与配置

**核心组件**: `GoogleAnalyticsInit.client.tsx`

- **动态加载 gtag.js**: 不硬编码 Measurement ID，而是从 `/api/public/company` 接口拉取各租户独立的 `gaMeasurementId`
- **持久化缓存**: ID 缓存到 localStorage，避免重复请求
- **可控初始化**: `send_page_view: false` —— 禁用 GA 默认自动上报，由我们手动控制 page_view 时机
- **调试模式**: URL 参数 `?gaDebug=1` 或 localStorage `ga-debug=1` 可开启详细日志

**组件位置**: `AnalyticsTracker.client.tsx`（全局挂载，监听路由变化）

### 2.2 核心指标：PV / UV / 活跃用户

| 指标 | 事件名 | 触发时机 | 去重机制 |
|------|--------|----------|----------|
| **PV** (页面浏览) | `gameline_page_view` + `page_view` | 路由变化时 | 每次路由变化都上报 |
| **UV** (独立访客) | `gameline_uv_visit` | 路由变化时 | 同路径同一天只上报一次（localStorage 标记） |
| **活跃用户** | `gameline_active_user` | 页面可见且聚焦时 | 每 60 秒心跳上报一次 |

**访客标识**: 使用 `crypto.randomUUID()` 生成唯一 `visitor_id`，持久化在 localStorage 中，用于串联用户行为。

**排除路径**: `/test`, `/test-node-ui`, `/theme-test`, `/studio/view` 等内部页面不纳入全局 PV/UV 统计。

### 2.3 iframe 内嵌场景 —— postMessage 转发机制

**问题背景**: 应用通过 `<gameline-page-view>` Web Component（iframe）嵌入时，第三方 Cookie 限制导致 iframe 内 gtag.js 无法直接发送 `collect` 请求到 Google Analytics。

**解决方案**:

```
iframe 内应用                    宿主页面
     │                              │
     │  检测到在 iframe 中           │
     │  (window.name === "gameline-page-view")
     │                              │
     │  postMessage({               │
     │    type: "GAMELINE:GA_EVENT",│
     │    payload: {                │
     │      measurementId,  ◄───────┼── 携带自己的 GA ID
     │      eventName,              │
     │      params                  │
     │    }                         │
     │  })                          │
     │ ─────────────────────────────>│
     │                              │
     │                              │  gtag("event", name, {
     │                              │    ...params,
     │                              │    send_to: measurementId  ◄── 精确路由
     │                              │  })
```

**关键设计**:
- iframe 不加载 gtag.js，仅通过 `postMessage` 通信
- 宿主页面的单一 gtag.js 实例处理多个 Measurement ID
- 使用 `send_to` 参数精确路由，避免事件串流到错误的 GA 属性

### 2.4 多 GA 实例共存

宿主页面和 iframe 可能使用不同的 GA Measurement ID（如 `G-HOST-12345` vs `G-IFRAME-67890`）。通过 `send_to` 参数实现:

- 不指定 `send_to` → 事件发送到**所有**已配置的 GA 实例（应避免）
- 指定 `send_to: "G-IFRAME-ID"` → 仅发送到该属性

## 三、埋点链路详述

以下是数据从产生到最终存储的完整流转路径，分场景说明。

### 3.1 GA 初始化链路

```
① 页面加载 / 应用挂载
    │
    ▼
② GoogleAnalyticsInit.client.tsx mount
    ├── 读取 /api/public/company 返回的 gaMeasurementId
    ├── 缓存到 localStorage（供下次快速读取）
    └── setGaMeasurementId(id) → ga.ts 全局状态更新
    │
    ▼
③ 当 measurementId 有效时
    ├── 注入 gtag.js 脚本 (<Script src="googletagmanager.com/gtag/js?id=...")
    └── 执行内联初始化代码
        ├── window.dataLayer = window.dataLayer || []
        ├── window.gtag = function(){ dataLayer.push(arguments) }
        ├── gtag('js', new Date())
        └── gtag('config', 'G-XXXX', { send_page_view: false })
    │
    ▼
④ GA 就绪，等待业务事件调用
```

### 3.2 页面访问链路（PV / UV / 活跃用户）

```
① 用户访问页面 / 路由变化
    │
    ▼
② AnalyticsTracker.client.tsx 监听 pathname + searchParams
    ├── 组装 fullPath（含 queryString）
    └── 判断是否在排除路径列表（/test, /studio/view 等）
    │
    ▼
③ 同步 host_app_id（来自 gameline-page-view 的 app-id）
    └── syncGaHostAppId() → ga.ts 状态更新
    │
    ▼
④ 触发核心指标（非排除路径）
    ├── trackPageView(fullPath)     → gameline_page_view 事件
    ├── trackUvVisit(fullPath)      → gameline_uv_visit 事件（日维度去重）
    └── sendDefaultPageView(fullPath) → GA 标准 page_view 事件
    │
    ▼
⑤ 进入活跃心跳循环
    ├── 页面可见 + 有焦点时：trackActiveUser(fullPath)
    └── setInterval(60s) 周期性上报 gameline_active_user
```

**UV 去重链路细节**:
```
trackUvVisit(path)
    │
    ├── ensureVisitorId() → 从 localStorage 读取或生成 visitor_id
    │
    ├── 构造 UV 标记键: "ga-uv-mark:{path}:{YYYY-MM-DD}"
    │
    ├── 检查 localStorage 是否已有该键
    │   ├── 存在 → 直接返回，不再上报
    │   └── 不存在 → 写入标记，继续上报
    │
    └── gtagEvent("gameline_uv_visit", { page_path, visitor_id })
```

### 3.3 业务事件链路（标准页面 —— 非 iframe）

```
① 用户交互（如：点击 Tab、添加购物车、搜索）
    │
    ▼
② 组件层调用封装函数
    ├── trackTabSwitch({ tab_group, from_tab, to_tab })
    ├── trackCartAdd({ item_id, order_type, ... })
    ├── trackSearch({ keyword, result_count })
    └── ...（20+ 个业务事件）
    │
    ▼
③ ga.ts 中的 gtagEvent(eventName, params) 统一处理
    ├── 检查 isGaEnabled() → measurementId 是否已配置
    ├── 检查是否在 iframe → isInPageViewIframe()
    │   └── 是 → 走【iframe 转发链路】（见 3.4）
    │
    └── 否 → 直接上报
        ├── 合并 baseGaEventParams() → host_app_id
        ├── 附加 send_to: measurementId（精确路由）
        ├── 附加 from_page（当前 pathname）
        └── 调用 window.gtag("event", eventName, payload)
    │
    ▼
④ gtag.js 内部处理
    ├── 将事件 push 到 dataLayer
    ├── 格式化后发送到 google-analytics.com/g/collect
    └── Network 面板可见 POST 请求
```

### 3.4 业务事件链路（iframe 内嵌场景）

```
① 用户交互（iframe 内）
    │
    ▼
② 组件层同样调用 trackXxx() 或 gtagEvent()
    │
    ▼
③ ga.ts 中 isInPageViewIframe() 返回 true
    │
    ├── 检测条件：window.self !== window.top && window.name === "gameline-page-view"
    │
    └── 进入 iframe 转发分支
        ├── 构造 postMessage payload
        │   ├── type: "GAMELINE:GA_EVENT"
        │   ├── payload.measurementId: 当前租户的 GA ID
        │   ├── payload.eventName: 业务事件名
        │   └── payload.params: { host_app_id, from_page, ...业务参数 }
        │
        └── window.parent.postMessage(payload, "*")
    │
    ▼
④ 宿主页面 PageViewElement 监听 message
    ├── 校验 type === "GAMELINE:GA_EVENT"
    │
    └── _forwardGaEvent(payload)
        ├── 解包 { measurementId, eventName, params }
        ├── 检查 gtag 是否就绪
        └── 调用 window.gtag("event", eventName, {
        │       ...params,
        │       send_to: measurementId    ◄── 精确路由到对应 GA 属性
        │   })
    │
    ▼
⑤ 宿主页面的 gtag.js 发送 collect 请求
    └── Network 面板在宿主页面可见请求
```

**iframe PageView 特殊链路**:
```
iframe 内路由变化
    │
    ├── sendPageViewToHost(path)
    │   └── postMessage({ type: "GAMELINE:GA_PAGE_VIEW", payload: { measurementId, path, extra } })
    │
    └── 宿主 PageViewElement._forwardGaPageView()
        └── gtag("event", "page_view", { send_to: measurementId, page_path: path, ... })
```

### 3.5 性能监控链路（DRT）

```
① 路由变化（Layout 层）
    │
    ▼
② markDRTStart(pathname)
    └── performance.mark(`drt-start-${pathname}`)
    │
    ▼
③ 页面组件挂载，发起数据请求
    │
    ▼
④ 数据加载完成（useDRT Hook 监听状态变化）
    ├── 条件检查：isSuccess && data && !isFetching && 未上报过
    │
    └── 满足条件 → measureAndReportDRT()
        ├── performance.mark(`drt-end-${pathname}`)
        ├── performance.measure(`drt-${pathname}`, startMark, endMark)
        ├── 计算 duration（毫秒）
        ├── getRating() 评级：good / needs-improvement / poor
        │
        └── runOnIdle() 空闲时上报
            └── fetch("/api/performance", {
                    method: "POST",
                    body: JSON.stringify({ metrics: [{ name:"DRT", value, rating, pathname, metadata }] }),
                    keepalive: true
                })
    │
    ▼
⑤ 服务端 /api/performance/route.ts 处理
    ├── 解析 JSON body
    ├── 遍历 metrics 数组
    ├── DRT 指标 → drtHistogram.labels(pathname, rating, source).observe(value / 1000)
    └── 写入 Prometheus metricsRegistry
    │
    ▼
⑥ Prometheus 抓取 → Grafana 展示 / 告警
```

### 3.6 数据最终流向对比

| 数据类型 | 采集端 | 传输方式 | 接收端 | 存储/分析 |
|----------|--------|----------|--------|-----------|
| **页面 PV** | AnalyticsTracker | gtag.js / postMessage | Google Analytics | GA4 实时报告 / 探索分析 |
| **UV / 活跃用户** | AnalyticsTracker | gtag.js / postMessage | Google Analytics | GA4 受众报告 |
| **业务事件** | 各业务组件 | gtag.js / postMessage | Google Analytics | GA4 事件报告 / 转化漏斗 |
| **DRT 性能** | useDRT Hook | fetch POST | /api/performance | Prometheus → Grafana |

## 四、前端性能监控体系

### 4.1 DRT (Data Ready Time) —— 关键数据就绪时间

**定义**: 从页面路由变化到首屏关键数据加载完成的时间。

**采集方式**:
```
Layout 层                    页面层
   │                            │
   │ markDRTStart(pathname)     │
   │ ─────────────────────────> │
   │   (performance.mark)       │
   │                            │ 数据加载完成
   │                            │ measureAndReportDRT()
   │                            │ ────────────────────>
   │                            │   performance.measure()
   │                            │   → 计算耗时 → 上报 /api/performance
```

**使用方式**:
```typescript
// 在页面组件中使用 Hook
import { useDRT } from "@/hooks/useDRT";

function MyPage() {
  const query = useQuery({ queryKey: ["data"], queryFn: fetchData });
  
  useDRT({
    isSuccess: query.isSuccess,
    data: query.data,
    isFetching: query.isFetching,
  });
}
```

**上报条件**（需同时满足）:
1. 数据加载成功 (`isSuccess === true`)
2. 数据存在 (`data` 不为空)
3. 不在获取数据中 (`isFetching === false`)
4. 当前路由还未上报过

**性能阈值**:
- 🟢 良好: ≤ 1500ms
- 🟡 需改进: 1500ms ~ 3000ms
- 🔴 较差: > 3000ms

### 4.2 服务端指标存储

上报到 `/api/performance` 的数据会写入 **Prometheus Histogram**:

```
render_engine_frontend_drt_seconds
  ├── labels: pathname, rating, source
  └── buckets: [0.1, 0.25, 0.5, 1, 1.5, 2, 2.5, 3, 5, 10]
```

## 五、组件间事件通信（EventBus）

### 5.1 设计定位

EventBus 用于**低代码场景下组件间的联动通信**，不是埋点上报通道，但与埋点体系同属事件驱动架构。

### 5.2 核心能力

| 能力 | 说明 |
|------|------|
| **发布/订阅** | `bus.publish(topic, payload)` / `bus.subscribe(topic, handler)` |
| **联动路由** | 基于规则自动将源组件事件转发到目标组件动作 |
| **条件过滤** | `when` 条件（`exists` / `equals`） |
| **负载映射** | `transform` 支持字段映射（`$.field` 语法） |
| **派发模式** | `sync` / `micro` / `timeout` |

### 5.3 典型联动规则

```typescript
// event-rules.ts
{
  sourceType: "menusGridTabsCollapsible",
  targetType: "sportList",
  sourceEvent: "menus.selectSport",
  targetAction: "changeSport",
  deliver: { mode: "micro" }
}
```

## 八、总结

| 维度 | 实现要点 |
|------|----------|
| **行为埋点** | 基于 GA4，统一封装 20+ 业务事件，自动处理 iframe 内嵌场景的跨域上报 |
| **核心指标** | PV（每次路由）、UV（日维度去重）、活跃用户（60s 心跳） |
| **性能监控** | DRT 指标覆盖主要页面，基于 Performance API，上报到 Prometheus |
| **组件通信** | EventBus 支持低代码场景下的组件联动，与埋点体系解耦 |
| **扩展性** | 租户级 Measurement ID、多 GA 实例共存、调试开关 |
